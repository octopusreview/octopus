# Security Policy

We take the security of Octopus seriously. This document describes how to report a vulnerability, what we treat as in scope, and what to expect after you tell us.

For the public-facing security overview (data flow, encryption, access controls, audit logging), see [https://octopus-review.ai/docs/security-overview](https://octopus-review.ai/docs/security-overview).
For the bug bounty programme and hall of fame, see [https://octopus-review.ai/docs/security](https://octopus-review.ai/docs/security).

## Reporting a vulnerability

Email **security@octopus-review.ai**. **Do not file a public GitHub issue.**

Please include:

- A clear description of the vulnerability
- Reproduction steps (proof-of-concept code is welcome)
- The affected component (web app, CLI, GitHub App, …) and version / commit if known
- Any relevant logs, screenshots, or recordings
- Your suggested fix or mitigation (optional)
- Whether you would like to be credited in the hall of fame

A PGP key for encrypted reports is published at https://octopus-review.ai/.well-known/security-public-key.asc (planned — until that lands, plain email is fine).

## Response SLAs

| Severity | First response | Initial assessment | Fix target |
|---|---|---|---|
| 🔴 Critical | 24 h | 48 h | 7 days |
| 🟠 High | 48 h | 5 business days | 30 days |
| 🟡 Medium | 5 business days | 10 business days | 90 days |
| 🔵 Low / Info | 10 business days | 20 business days | best-effort |

These are upper bounds. Most reports get a first reply much faster.

## Disclosure timeline

We follow [coordinated disclosure](https://www.iso.org/standard/72311.html):

1. You report privately.
2. We acknowledge, triage, and reproduce.
3. We develop a fix; we may ask you to validate it.
4. We deploy to hosted Octopus and publish a patched release for self-hosters.
5. We publish a security advisory crediting you (if you opted in) **at least 30 days** after the fix is available — earlier with your consent.

If we cannot fix a vulnerability within 90 days of acknowledgement, we will agree a coordinated extension or publish jointly with you.

## Scope

**In scope:**

- The web application at `octopus-review.ai` and its API endpoints
- The Octopus GitHub App, GitLab integration, and Bitbucket integration
- The `@octopus/cli` binary distributed via curl/irm install scripts
- The official self-hosting Docker images at `ghcr.io/octopusreview/octopus`
- Webhook verification flow (HMAC validation, replay resistance)
- Authentication flow (Better Auth, device-code, OAuth providers)
- Prisma migrations and database schema integrity

**Out of scope:**

- Third-party vendor vulnerabilities — please report those upstream. We will follow up if we use a vulnerable version.
- Findings that require physical access to a user's machine
- Denial-of-service attacks that don't require an authentication or privilege flaw
- Self-XSS that requires the victim to paste attacker-controlled content into the dev tools
- Missing security headers without a demonstrated attack chain
- Issues solely affecting outdated browsers (more than two majors behind the current Chrome/Firefox/Safari)
- Reports generated by automated scanners without a demonstrated impact
- Social engineering attacks against Octopus staff
- Forks of Octopus (the modified-MIT licence allows them, but we don't own their deployments)

## Safe harbour

We will not pursue civil or criminal action against researchers who:

- Give us reasonable time to fix issues before public disclosure
- Make a good-faith effort to avoid violating user privacy, destroying data, or interrupting service
- Do not exfiltrate user data beyond what is strictly necessary to demonstrate the vulnerability
- Do not exploit vulnerabilities for personal gain

If you are unsure whether your testing fits within safe harbour, contact us first and we will tell you.

## Self-hosting security best practices

When self-hosting Octopus, please ensure:

- All API keys and secrets are stored securely and never committed to version control
- The `.env` file is properly configured and excluded from git
- Database access is restricted to trusted networks
- HTTPS is enabled in production (terminate at your reverse proxy or use the bundled nginx config in `infra/nginx/`)
- GitHub/Bitbucket/GitLab webhook secrets are configured to verify payload authenticity
- The container runs as a non-root user
- Backups of `postgres_data` and `qdrant_data` volumes are encrypted at rest
- The `OCTOPUS_SERVER_ID` env is unique per instance for audit traceability

## Supported versions

| Version | Supported | Notes |
|---|---|---|
| Latest hosted Octopus | ✓ | Always patched |
| Latest tagged release on GHCR | ✓ | Self-hosted users should upgrade within 30 days of a security release |
| Older tagged releases | ✗ | Upgrade to the latest tag — we do not backport |
